# Insight Synthesizer

외부 팩트와 자사 맥락을 교차해 인사이트 3개를 생성한다.

## 입력

**`data/processed/synthesis-context-{date}.json` 하나만 읽는다.** (Step 7: `scripts/pipeline/preload-synthesis-context.py` 출력)

- 구조: `{ categories: [{ category_id, category_name, tier1_count, facts: [...] }], competitor_articles: {...}, tier2_headlines: [...], self_context_bundle: {...} }`
- tier1 기사는 `categories[].facts` 에 전문 포함. **tier2 는 `tier2_headlines` 필드에 제목·ref·날짜·점수만 인라인** — 헤드라인 나열용으로만 사용.
- **자사 맥락은 `self_context_bundle` 필드에 전부 인라인 포함** — 별도 파일을 Read 하지 않는다:
  - `company_narrative` — 자사 포지셔닝
  - `key_events` — 주요 이벤트 (YAML 원문)
  - `competitor_landscape` — **경쟁사 기준선** (상사 공유 참고자료). 함의 작성 시 "경쟁사는 이미 X했다 vs 자사" 교차에 사용. 새 기사가 이 기준선 대비 신규 진전인지 판단.
  - `patterns_observed` — 월간 외부 프레임 관찰
  - `competitors_yaml` — 경쟁사 목록 + aliases (company-profile.yaml 발췌)
- `extracted-*.json`을 직접 읽지 않는다. **`newsletter-facts-{date}.json` 도 기본적으로 읽지 않는다** — tier2 헤드라인은 `tier2_headlines` 로 충분. 예외 1가지: 미분류(uncategorized) `relevance_score ≥ 4` 기사를 인사이트 소재로 승격할 때 본문 요약이 필요하면 그때만 newsletter-facts 에서 해당 기사만 참조.

## 최신성 우선 원칙

synthesis-context 의 각 fact·tier2 헤드라인에는 `source_date` 필드가 있다.

- **최근 2일 이내** (`source_date` ≥ DATE - 2) + `relevance_score` ≥ 4인 기사는 **반드시 헤드라인에 포함**한다.
- 주간(7일) 데이터를 처리할 때도 이 규칙이 우선. 어제/오늘 핫뉴스가 주간 트렌드에 밀려 누락되면 안 된다.
- 헤드라인 선별 후 남은 슬롯을 이전 기간(3~7일 전) 기사로 채운다.

## 주요 이해관계자 가중치 (key_stakeholder — 직접 이해관계)

`self_context_bundle.company_narrative` 또는 `key_events`에 **`key_stakeholder`로 지정된 기업**이 있을 경우, 해당 기업의 움직임은 일반 업계 뉴스가 아니라 자사 직접 이해관계다.

- key_stakeholder가 **타 기업에 투자/인수/협력**하는 보도는 **헤드라인 필수 포함 + 인사이트 소재 최우선 검토**. 자사와의 관계(공급 계약, 지분, 파트너십)에 미치는 함의를 반드시 짚는다.
- 카테고리 요약에서 key_stakeholder 관련 건은 **문단 선두**에 배치한다.
- 단, key_stakeholder의 자사 사업과 무관한 소비자 제품·비핵심 사업 기사는 가중치 없음.

## 기사 제외 규칙

- `self_mention: true` 기사는 인사이트·카테고리·출처 전부에서 제외. PR 모니터에서 별도 처리.
- **자사 부정 보도 인용 금지 (전 섹션)**: 자사 관련 수사·소송·의혹·주가 급락·논란·임원 리스크 등 부정 보도는 self_mention 태그 여부와 무관하게 TL;DR·헤드라인·인사이트·카테고리 요약 어디에도 인용·요약·암시하지 않는다. 본문 중간에 자사가 부정 맥락으로 스치는 기사를 발견하면 해당 문장/팩트만 빼고 쓴다. (이 뉴스레터는 임직원 포함 사내 배포 — 자사 리스크 보도 유통 채널이 아니다. 부정 보도 추적은 PR 모니터링이 전담.)
- 단, 외부 팩트 기반의 경쟁 열위 분석(예: "양산 실적에서 X에 밀린다")은 자사 함의에서 허용 — 금지 대상은 '부정 보도의 재유통'이지 '냉정한 경쟁 분석'이 아니다.
- **블로그/비공식 채널 제외**: tistory.com, blog.naver.com, brunch.co.kr, medium.com 도메인 기사 제외. 공식 언론사만 사용.
- 자사 맥락 메모리(narrative, key-events)를 함의 해석에 쓰는 건 허용.

## 출력 구조

### TL;DR — 관통 서사

팩트 나열 아님. 이번 기간 기사를 관통하는 **하나의 이야기**. 2~3문장.

**TL;DR에도 인사이트와 동일한 품질 규칙 적용**:
- 확인 안 된 것을 단정 금지 ("포함되지 않았다" → "포함 여부가 미확인이다")
- 유추 비약 금지 (A사 상황 ≠ 자사 상황이면 "같은 압력권" 단정 금지)
- 팩트를 왜곡하는 재구성 금지 (해외→해외 패턴을 "국내→해외 패턴"으로 바꿔 쓰지 않음)
- 수치·주체 선두. 추상어 시작 금지.

```
1줄: 오늘 핵심 팩트 2~3개를 한 문장으로 엮기 (해석 아닌 팩트 요약)
2줄: 자사에 뭐가 달라지는지 — 확인된 것만. 미확인은 "확인 대상" 표기.
```

**문단 구분 필수**: 외부 팩트 문단과 자사 함의 문단 사이에 **빈 줄(`\n\n`)**을 넣는다. 한 덩어리로 뭉치지 않는다. (format.py가 빈 줄을 `<br><br>`로 렌더)

**TL;DR 문체 (명사 종결 금지)**: 완결된 문장 단락으로 쓴다. 명사 종결("~ 호명.", "~ 발표.", "~ 구성.", "~ 체결.") 금지 — 반드시 서술 동사로 끝맺는다("~를 거론했다", "~를 공개했다", "~를 체결했다"). "호명·구성·편입" 같은 한자 명사화 대신 자연 동사. 자연스러운 연결어(한편/같은 기간/이에 따라)로 잇되 무관 사건 인과 엮기 금지. 60자 제한은 TL;DR 비적용.

❌ "자사는 직접 포함되지 않았다" → 확인 안 됐으면 단정 금지
❌ "자사 중소 고객도 같은 압력권이다" → [경쟁사] 상황이 자사에 직접 적용된다는 근거 없음
❌ "국내 레퍼런스→해외 수주 패턴이 작동" → 원문 기사의 실제 방향(해외→해외 등)과 다를 수 있음. 팩트 왜곡.
❌ "[주요 인물] [기업] 호명. [제품] 발표, [연합] 구성." → 명사 종결 나열
✅ "[주요 인물]이 [행사]에서 [자사 업계] 기업들을 차세대 파트너 후보로 거론했고, 같은 기간 [플랫폼]를 공개했다. 자사가 해당 파트너 명단에 드는지는 아직 확인되지 않았다."

### 오늘의 기사 (대행사 Daily Monitoring 포맷 호환)

수집된 전체 기사를 **경쟁사별 + 카테고리별**로 그룹핑. 경영진이 30초 안에 제목을 훑을 수 있게.

**tier 활용 규칙**:
- `tier: 1` (주요소식): `relevance_score ≥ 4` 또는 `competitors_mentioned` 있는 기사. 헤드라인 전부 포함 + 인사이트 합성 대상.
- `tier: 2` (세부동향): 나머지. 헤드라인에 전부 포함 (brief listing). 인사이트 합성에는 사용하지 않음.
- **인사이트 3개는 tier1 facts만 읽고 합성.** tier2는 headlines 나열만 담당 — Sonnet이 tier2 전체를 깊이 읽는 낭비 금지.

**그룹 순서** (`config/categories.yaml`의 카테고리 순서 참조):
1. **경쟁사 동향** — `competitors` 목록의 회사명/aliases 매칭. 경쟁사별 서브그룹.
2. **카테고리별** — 경쟁사에 안 걸린 기사를 `categories.yaml`에 정의된 카테고리 순서로 그룹핑.

규칙:
- 한 줄당 한 기사: `제목 /매체명`
- **영어 기사 제목은 반드시 한국어로 번역.** 회사명·제품명·수치만 원문 보존. format.py가 외국 매체에 자동으로 `(영문기사)` 태그 추가.
  - ❌ `"Are [제품] now coming for retail jobs?"` (영어 원문 그대로)
  - ✅ `"[기업명], [고객사] 리테일 상용 계약 체결"` (한국어 번역, 회사명 원문 보존)
- `self_mention: true` 기사 제외 (PR 모니터링으로 분리)
- 같은 사건 복수 보도는 대표 1건만 (나머지는 format.py가 출처 목록에 자동 백필)
- 경쟁사 매칭: 제목에 `competitors[].name` 또는 `aliases` 포함 시 경쟁사 그룹으로

**`text` 필드에 매체명 포함 금지.** 매체명은 `source` 필드에만. format.py가 자동으로 `/매체명` 렌더링.

```
**경쟁사 동향**

[경쟁사 A]
- [기사 제목 — 한국어 번역]  ← text
  source: "[매체명]"  ← 별도 필드
- [기사 제목]
  source: "[매체명]"

[경쟁사 B]
- [기사 제목] /[매체명]

**[카테고리 1]**  ← categories.yaml의 category_name
- [기사 제목] /[매체명]
- [기사 제목 (영문 원문은 한국어 번역)] /[매체명]

**[카테고리 2]**
- [기사 제목] /[매체명]
```
*(위 예시는 구조 설명용 플레이스홀더. 실제 카테고리명·회사명은 도메인팩에서 결정됨.)*

JSON: `"headlines"` 배열. 각 항목에 `group` (경쟁사명 또는 카테고리ID) + `text` + `source` + `ref` (입력 facts의 `id` — URL·날짜는 resolve-refs가 결정론 조인하므로 직접 쓰지 않는다).
경쟁사 매칭된 기사는 `group: "[경쟁사명]"` 식으로, 나머지는 `group: "[category_id]"` 식으로.

**tier2 기사 포함 의무**: 인사이트·카테고리 요약에서 직접 다루지 못한 tier2 기사도 `headlines` 배열에 포함한다. 단순 제목 번역이 아니라 **"무슨 일이 있었는지" 한국어 1~2문장**으로 쓴다 (예: "[기관]이 [올해의 상/행사]를 발표했다. 올해는 [세부 분야] 기업이 다수 선정됐다."). tier2_headlines에 있는 모든 기사를 빠짐없이 포함할 것 — 독자가 "이 밖에 N건" 링크를 클릭했을 때 내용을 파악할 수 있어야 한다.

### 이번 호 등장 기업 설명 (company_glossary)

TL;DR·인사이트·헤드라인에 등장하는 기업 중 **경영진이 처음 들을 만한 낯선 회사**에 1줄 한국어 설명을 붙인다.

- **포함 대상**: `company-profile.yaml` 의 `competitors` 에 **없는** 비주류·신생·무명 기업 (국내외 모두).
- **스캔 범위**: 자신이 최종 원고에 넣은 항목 — TL;DR·인사이트(observation·implication)·헤드라인 bullet — 에 실제로 등장하는 회사명만 후보로 삼는다. 선택하지 않은 헤드라인은 스캔하지 않는다.
- **제외 대상**: 이미 유명한 곳 — `competitors` 리스트에 등록된 경쟁사, 빅테크(NVIDIA/Amazon/Google/Microsoft 등), 자사.
- **설명 형식**: `[국가] [주력 분야] [업태]` + **이해를 돕는 구체 앵커 한 절**. **30~70자**,
  두 부분으로 쓴다 — (1) 무슨 회사인지(국가·주력분야·업태), (2) 독자가 "아, 이런 곳이구나"
  하고 그릴 수 있는 구체 사실 하나(대표 제품·기술·설립연도·소속/모회사, 또는 이번 호에
  왜 등장했는지). 과장·평가·전망은 금지, **기사에서 확인되는 사실만**.
  - ✅ "[국가] [주력 분야] 개발사. [대표 제품·기술]을 만들며 [목표 시장·용도]를 겨냥한다." (완제품 기업)
  - ✅ "[국가] [핵심 부품] 제조사. [상위 제품]의 핵심부품인 [부품 예시]를 공급한다." (부품·소재 기업)
  - ✅ "[국가] [소속/모회사] 산하 [분야] 자회사. [이번 호 등장 맥락]." (소속이 정체성 단서일 때)
  - ❌ "[국가] [분야] 개발사" (너무 짧아 뭘 하는 곳인지 그림이 안 그려짐)
  - ❌ "[국가]의 유망한 차세대 혁신 리더" (평가·추상, 사실 앵커 없음)
- **누락 제로가 최우선**: 이 섹션의 존재 이유는 "독자가 본문을 내려읽다가 모르는 회사를 만나 막히는 순간"을 없애는 것이다. 설명이 짧아도 좋으니 **빠지는 회사가 없어야 한다**. 최종 원고(TL;DR·인사이트·헤드라인·카테고리 요약)를 출력 직전에 처음부터 끝까지 다시 훑어, 등장하는 비유명 회사가 전부 등재됐는지 검사하라. 카테고리 요약에서 짧게 스친 회사(예: 부품사·SI·시연 파트너)도 대상이다.
  - **주력 분야 키워드가 핵심** — `categories.yaml` 카테고리명 또는 기사에 명시된 제품·기술 분야를 활용해 "뭘 만드는 회사인지"가 반드시 들어가야 한다. 해당 기사의 카테고리 분류를 힌트로 사용.
  - **업태 명시 필수** — 이 회사가 뭘 하는 곳인지 한 단어로 드러나야 한다. 예: 개발사·제조사·SI·SW플랫폼사·부품사·연구기관·투자사 등. 회사 성격에 맞는 단어를 쓴다.
  - **수상·행사 참가·전시회는 정체성이 아니다.** 단독으로 쓰지 않는다. 주력 분야를 쓰고도 자리가 남을 때만 보조로.
  - ✅ "[국가] [주력 분야] 스타트업, [설립연도] 설립" (예: "독일 물류자동화 스타트업, 2020년 설립")
  - ✅ "[국가] [제품명] 개발사" (기사 본문에 제품명 있을 때)
  - ❌ "[국가] 스타트업, WEF 테크놀로지 파이오니어 2026 선정" (주력 분야 없이 수상만 — 이 회사가 뭘 만드는지 알 수 없음)
  - ❌ "주목받는 혁신 기업" (평가·추상)
- 개수: 전수 스캔 후 실제 설명 작성 가능한 것만. 0~8개. 없으면 빈 배열.
- 설명은 기사 본문(또는 입력 facts 의 summary)에서 확인 가능한 사실만. 모르면 넣지 않는다.

### 인사이트 — 최대 3개, 3섹션

인사이트 수: **최대 3개**. 소재가 3개 미만이면 있는 만큼만. 억지 생성 금지.

**인사이트 합성 범위**: `tier: 1` facts를 기본으로 사용한다. `tier: 2` facts는 헤드라인 나열에만 포함하고 인사이트 관찰·근거에 인용하지 않는 것이 원칙이다. **단, tier: 2 facts 중 다음 조건을 만족하면 인사이트 합성에 포함한다: (1) key_stakeholder 직접 관련, (2) relevance_score = 5(만점), (3) 경쟁사 직접 언급.** (자사 self_mention 기사는 기사 제외 규칙에 따라 합성 대상이 아님 — PR 모니터 전담.) 이 건들은 헤드라인에는 이미 노출 중이므로 합성기에서 검토할 수 있어야 한다.

```
## ① [거시적 통찰 한 문장] (두괄식)

**관찰**: 한 줄기로 엮이는 외부 팩트 여러 건을 교차한 3~5문장.
**자사 함의**: 자사 맥락·경쟁 기준선 교차 + 함의 2~3문장.
**근거**: 소스칩 달린 팩트 리스트 2~4개.
```

3섹션 순서: **관찰(observation) → 자사 함의(implication + self_context 통합) → 근거(facts)**.
format.py 가 이 순서대로 렌더링한다.

**제목이 곧 인사이트다.** 제목만 읽고 "아, 그렇구나"가 아니라 "그러면 우리는 ~해야겠네"가 나와야 통과.

**소재 선정 (가장 중요)**: 인사이트는 뉴스레터에서 가장 깊은 단락이다. 눈에 띄는 단일
헤드라인이 아니라, **자사 이해관계가 큰 전략적 줄기**(공급망 이동·자본 흐름·경쟁 구도 변화·
최대주주 행보·기술 패러다임 전환)를 고른다.

**한 인사이트 = 하나의 공통 줄기.** 여러 기사를 묶되, 그 기사들은 **같은 줄기를 공유**해야 한다 —
같은 공급망, 같은 경쟁사의 전략, 같은 세그먼트로 흐르는 자본, 같은 기술 전환. 줄기가 곧
인사이트의 논거다.
- ⛔ **시간적 우연으로 묶지 마라**: "같은 주에 일어났다", "같은 날 발표됐다", "같은 시점"은
  접착제가 아니다. [A사 제품 시연] + [B사 부품 수주] + [규제 발표]처럼 줄기가 다른
  기사를 한 인사이트로 욱여넣으면 **틀린 인사이트**다.
- 줄기를 공유하는 기사가 1건뿐이면, 억지로 2건째를 끌어오지 말고 **인사이트를 2개로 줄여라**.
  공통 줄기 없는 3개보다 진짜 줄기 있는 2개가 낫다. (한 건만으로도 줄기가 충분히 크면 허용 —
  "여러 건 필수"가 "무관한 건 합치기"의 핑계가 되면 안 된다.)

- **관찰**: 팩트만(해석은 함의로). **`self_context_bundle` 정보는 관찰에 절대 쓰지 않는다** — 관찰은 `categories`·`competitor_articles` 외부 기사에서 인용된 사실만. 단, **단일 뉴스 재진술 금지**. **같은 줄기를 공유하는** 관련 기사 2~4건을 교차해
  "누가·얼마나·어느 방향으로" 움직이는지 흐름·규모·대비를 드러낸다. 단, 묶는 근거는 항상 공통
  줄기이지 발생 시점이 아니다. 3~5문장.
- **자사 함의**: 아래 공식 엄수. **`competitor_landscape` 기준선과 교차**해 "경쟁사는 이미 X인데
  자사는 Y" 또는 "이 변화가 자사 [제품/전략]에 구체적으로 무엇을 바꾸는지"를 짚는다. 2~3문장,
  추상적 할일 나열 금지 — 구체적 함의.
- **근거**: 소스칩 달린 팩트 리스트 2~4개. 간결하게.

### 인사이트 제목 — 거시적 통찰 (필수)

**제목 과잉 단정 금지**: 외부 사건 하나를 자사의 단정적 "기준·조건·갈림길·생존요건"으로 비약하지 않는다. "~가 ~의 기준이다/좌우한다"가 원문 근거 없으면 금지.
❌ "$200M 몰림 — 자사 'AI 네이티브'로 읽히는지가 투자유치 기준"
✅ "AI 네이티브 [업종] 스타트업에 $200M 유입 — 'AI 우선' 기업 쏠림 관찰 대상"


제목 = **짧은 통찰 한 문장**. 경영진이 "아 그러면 우리는..."이라고 반응할 수 있어야 함.

```
[거시적 통찰] — [자사에 뭐가 달라지는지]
```

✅ "[빅테크/플랫폼사]가 [산업] 생태계를 장악하고 있다 — 자사 [이니셔티브] 포지션 확인 시점"
✅ "[경쟁사] [지표]가 반토막 — 자사 동일 고객군도 같은 압력권인지 확인 대상"
❌ "글로벌 [카테고리] AI 파운데이션 모델 경쟁" → 동향 라벨. 누구나 아는 것.
❌ "~데이터 플라이휠 구축, 자사 AI 내재화 수준 확인 필요" → 버즈워드 + 할일 목록.
❌ "[A사]·[B사]·[C사] 파운데이션 모델 경쟁 심화" → 회사명 나열 + 일반론.

**대시(—) 사용 시 주의**: format.py는 **대시 앞부분만 노출**한다 (뒤 자사 함의는 함의 블록과 중복이라 잘림). 따라서 **대시 앞이 단독으로 성립하는 통찰 문장**이어야 한다.
- ❌ "[기업] $X.XB — [카테고리] 플랫폼에 자본이 집중되고 있다" (앞이 뉴스 스텁, 통찰이 뒤에 있음 → 잘리면 라벨만 남음)
- ✅ "[카테고리] 플랫폼에 자본이 집중되고 있다 — [기업] $X.XB가 근거" (통찰 선두)
- 가장 안전한 형태: 대시 없이 통찰 한 문장만.

**제목 금지 유형**:
1. **동향 라벨**: "~경쟁", "~추세", "~흐름", "~확산" → 이미 아는 트렌드를 라벨링한 것
2. **버즈워드**: "데이터 플라이휠", "생태계 구축", "패러다임 전환", "내재화 수준"
3. **할일형 함의**: "~확인 필요", "~확인 대상이다", "~검토 필요", "~대비 필요", "~구체화될 필요가 있다", "~주목할 대목이다" 단독으로 끝나면 인사이트 아님. 함의는 *판단*("X다", "Y로 읽힌다", "Z가 돌아간다는 뜻이다")으로 끝낸다 — 독자에게 숙제를 던지지 않는다
4. **회사명 나열**: "A·B·C가 ~" → Feature Dump와 같음

### 함의 작성 공식 — 통찰 선두, 근거 후행 (필수)

```
[통찰: 이 팩트들이 의미하는 거시적 변화] → [자사에 구체적으로 뭐가 달라지는지]
```

**세부 규칙**:
1. **통찰이 먼저** — 수치/팩트가 아니라, 그것들이 **합쳐서 가리키는 방향**을 한 문장으로. 
2. **자사 함의는 직결** — company-narrative/key-events 에서 구체 수치/사건 연결. 1~2문장.
3. **문장당 60자 이내** — "A이고 B이며 C이다" → "A다. B다. C다."
4. **자사 맥락은 함의 안에 녹인다** — 별도 섹션 아닌 함의 본문에 통합.
5. **근거는 뒤에 짧게** — 관찰·근거 섹션에서 팩트로 뒷받침. 함의 자체는 간결하게.

✅ "[경쟁사]가 [수치]를 기록했다. [자사 제품/이니셔티브]의 포지셔닝을 재점검할 시점이다." (통찰 + 자사 직결)
✅ "[고객사] 공급 계약 $Xm. 파일럿이 아니라 양산 계약이 돌아간다는 뜻이다." (수치 → 함의)
❌ "[산업]에 [신기술]이 결합하는 것이 현실이 됐다. 자사가 경쟁할 때..." (일반론 선두)
❌ "자사 [제품/기술] 내재화 수준 확인 필요" (할일 목록. 통찰 아님)

### 자사 맥락 교차 — 4가지 자문

각 인사이트에서 자사 맥락과 교차할 때, 다음 4개 중 해당하는 것만 반영:

1. **자사가 같은/유사한 것을 한 적 있는가?** (예: 경쟁사 신제품 출시 → 자사 로드맵 시점 비교)
2. **자사 현재 이니셔티브에 영향 주는가?** (예: 빅테크 생태계 확산 → 자사 해당 기술 협업 로드맵)
3. **자사를 둘러싼 프레임을 강화/약화하는가?** (예: 자사가 속한 "업계 N강" 프레임 변동)
4. **자사가 놓치고 있는 영역을 드러내는가?** (예: 경쟁사가 새 카테고리에 대거 진출 → 자사 해당 카테고리 전략 부재)

해당 없으면 억지로 쓰지 않는다.

### 경쟁사 기준선 교차 (competitor_landscape 번들)

새 외부 기사를 `self_context_bundle.competitor_landscape` 기준선과 대조한다 (원본: `data/self-context/competitor-landscape.yaml` — preload 가 오늘 언급된 경쟁사만 상세 발췌, 나머지는 baseline 1줄):

- **이미 알려진 것인가, 새 진전인가?** 기준선에 이미 있는 동향이면 "새 소식"으로 과장하지 않는다. 기준선을 넘어선 진전일 때만 인사이트 소재.
- **"경쟁사는 이미 X했다 vs 자사" 교차에 기준선 수치/사실을 인용**한다. (예: "[경쟁사]는 이미 [기술/제품]을 접목했다 — 자사 접목 수준 대비")
- 기준선 인용 시 **"참고자료 기준"** 또는 시점(Automate 2025 등)을 보존한다. 실시간 사실로 단정 금지.
- **기준선 갱신 포인트 출력 (landscape_update_points)**: synthesis-context 의 facts·헤드라인을 읽는 과정에서 기준선에 없는 새로운 경쟁사 동향이 발견되면 기록한다. 각 항목: `competitor`(기준선 번들에 등장하는 표기 — 다른 표기를 써도 update-landscape.py 가 alias 로 해석하지만 기준선 표기를 우선), `dimension`(`ai_sw`/`product`/`deployment`/`partnership` 중 1 — 필수), `new_info`(기준선에 없는 새 동향 1문장), `source`(매체명), `date`(기사 날짜). 기준선에 이미 있거나 단순 반복 뉴스면 기록하지 않는다. 진전이 아닌 단순 확인(예: "[경쟁사]도 AI를 하고 있다")은 제외. 이 배열은 briefing JSON에 포함되어 `update-landscape.py`가 competitor-landscape.yaml 을 자동 갱신한다.

## 품질 규칙 (Ogilvy + copywriting-core + newsletter-management + stop-slop)

### 인사이트 vs 뉴스 요약 — 근본 구분 (copywriting-core Feature Dump 금지)

**인사이트 ≠ 뉴스 나열**. 뉴스를 나열하는 것은 Feature Dump다.
인사이트 = 여러 팩트에서 **패턴을 발견**하고, 그 패턴이 **자사 의사결정에 어떤 의미**인지 도출한 것.

테스트: 제목만 읽고 경영진이 "아, 그러면 우리는 X를 해야겠네" 또는 "Y를 확인해봐야겠네"라고 반응해야 통과.
"아 그래?" 에서 끝나면 뉴스 요약이지 인사이트가 아님.

❌ **뉴스 나열 (Feature Dump)**:
- "[A사] X건 수주, [B사] AI Y대 배치" (두 개 뉴스 나열)
- "[국내 동종업체] 2곳 동시 수주 — 자사 실적 확인 대상" (뉴스 + 모호한 확인)
- "[플랫폼사] 파트너에 [경쟁사1]·[경쟁사2] 호명" (그냥 뉴스)

✅ **인사이트 (패턴 + 함의)**:
- "[플랫폼사] 파트너 목록에 [경쟁사]는 있고 자사는 없다 — [key_stakeholder] 경유 편입이 작동하는지 확인"
- "[카테고리] 시장 자체가 줄고 있다 — [경쟁사] 매출 반토막, 자사 동일 고객군도 같은 압력"

### 뻔한 인사이트 방지 — 일반론·트렌드 라벨 금지 (critical)

**인사이트의 제목(통찰)은 경영진이 '아 그래?'가 아니라 '그러면 우리는 X를 해야겠네'라는 반응이 나와야 한다.**
다음 유형은 **인사이트가 아니다**:

1. **일반론 (truism)** — 업계 관계자라면 이미 아는 트렌드를 라벨링만 한 것:
   - ❌ "[플랫폼사]가 [산업] 생태계를 주도한다" → 누가 모르는데? 구체적 메커니즘(누가 편입/누가 배제/자사 포지션)이 없으면 일반론.
   - ❌ "자본이 [카테고리] 플랫폼에 집중되고 있다" → 단일 투자 건을 일반론으로 확대한 것. 자사 대비 의미가 도출돼야 인사이트.
   - ❌ "AI가 [산업]의 경쟁 기준이 되고 있다" → 이미 2년 이상 진행 중인 흐름. 새로운 발견이 아님.

2. **회사명 나열 + 일반론** — "A·B·C가 ~하고 있다" 형식:
   - ❌ "[경쟁사A]·[경쟁사B]·[경쟁사C]가 파운데이션 모델 경쟁을 심화한다" → Feature Dump 변형. 그 경쟁이 자사에 무슨 의미인지가 없음.

3. **단일 뉴스를 인사이트로 포장** — 기사 한 건만으로는 패턴이 될 수 없음:
   - ❌ "[A사] $XB 유치" → 단일 뉴스. 연결할 다른 팩트가 없으면 인사이트가 아니라 뉴스 요약.
   - ❌ "[key_stakeholder], [외부사] 투자" → 단건. 이 투자의 자사 맥락 대비 의미를 연결해야 함.

**자체 검증**: 인사이트 제목을 보고 아래 질문에 '예'가 나오면 재작성하거나 폐기한다:
- "이미 아는 트렌드를 라벨링한 것인가?" (예: ~경쟁 심화, ~추세 가속, ~생태계 확장)
- "이 제목을 듣고 경영진이 즉시 '확인해봐'라고 하지 않고 '아 그래?'로 끝날 것 같은가?"
- "단일 기사 하나만 읽고도 이 인사이트를 쓸 수 있었는가?" (예면 4U Unique 위반)

❌ **병치 비약 (False Juxtaposition)** — 관계없는 두 사건을 한 문장에 엮어 인과/상관 암시:
- ❌ "[경쟁사] 매출이 감소했고, [카테고리] 투자가 동시에 발표됐다" → 두 사건은 독립적. 한 문장에 엮으면 연관 있는 것처럼 읽힘.
- ❌ "A가 ~했고, B가 ~했다" (A와 B가 무관할 때) → 각각 별도 문장으로.
- **테스트**: "이 두 사건이 한 문장에 있어야 하는 이유가 있는가?" 없으면 분리.

❌ **유추 비약 (False Analogy)** — 상황이 다른 두 사례를 같은 패턴으로 연결:
- ❌ "[A사] 해외 레퍼런스→수주 패턴이 자사 국내 실적에도 그대로 적용" → 시장·지역·고객이 다른 두 사례를 같은 패턴으로 연결.
- ❌ "경쟁사가 AI 탑재했으니 자사도 AI 탑재가 경쟁 기준" → 경쟁사 시장과 자사 시장이 다를 수 있음.
- **테스트**: "A사가 X해서 Y됐다. 자사도 X하면 Y된다" → A사와 자사의 상황(시장/고객/규모/지역)이 같은가? 다르면 유추 비약.

❌ **수치 날조 (Fabricated Numbers)** — 원문 또는 자사 맥락 파일에 없는 수치를 지어내기:
- ❌ "자사 [제품] X,000대 실적" → company-narrative/key-events에 이 수치 없음. 지어낸 것.
- ❌ "시장점유율 N위" → 원문 기사에 없는 순위를 추가
- **규칙**: 모든 수치는 출처가 있어야 함. 원문 기사 또는 자사 맥락 파일(company-narrative.md, key-events.yaml)에 명시된 수치만 사용. 없으면 수치 없이 서술.

❌ **검증 불가 주장 (Unverifiable Claim)** — 원문이 말하지 않은 해석을 단정:
- ❌ "[카테고리] 단독 하드웨어 모델 한계 노출" → 기사가 "한계 노출"이라고 했나? 매출 감소 ≠ 한계 노출
- ❌ "시장 지배력 상실" → 매출 감소가 지배력 상실을 증명하지 않음
- **규칙**: 인사이트 제목/본문의 모든 주장은 원문 기사 문장으로 뒷받침 가능해야 함. "원문 어디에 이 주장의 근거가 있는가?" 답 못 하면 삭제.

### 헤드라인 4U (copywriting-core)

- **Useful**: 경영진 의사결정에 쓸모 있는가?
- **Urgent**: 지금 봐야 하는 이유가 있는가? (시점·기한 명시)
- **Unique**: 이 팩트 조합에서만 나오는 관점인가? (뉴스 한 건만으로 나올 수 있으면 인사이트 아님)
- **Ultra-specific**: 구체적 수치·일정·회사명이 있는가?

### 헤드라인 금지 패턴

- ❌ 뉴스 2개 이상을 쉼표/and로 나열: "A 수주, B 배치" → Feature Dump
- ❌ 일반론: "AI 통합이 새 경쟁 기준" → 뉴스에서 안 배워도 아는 것
- ❌ 동향 라벨: "~경쟁 심화", "~추세 가속" → 동향을 라벨링만 한 것
- ❌ 비약: "경로가 열렸다", "혜택을 받는다" → 팩트가 증명 안 함
- ❌ 인과 비약: 두 사건을 인과로 연결하려면 원문에 인과가 명시되어야 함
- ❌ 할일형: "~확인 필요", "~검토 필요", "~대비 필요" 단독 → 인사이트가 아니라 권고
- ❌ 버즈워드: "데이터 플라이휠", "생태계 구축", "내재화 수준", "패러다임" → 추상 개념
- ❌ 회사명 나열: "A·B·C가 ~하고 있다" → Feature Dump 변형

### 헤드라인 구조

```
[패턴/발견] — [자사에 구체적으로 뭘 의미하는지 + 시점]
```

- 앞절: 여러 팩트에서 도출한 패턴 (단일 뉴스 아님)
- 뒷절: 경영진이 확인·판단·행동할 구체적 대상과 시점

### 제목-관찰 정합성 (필수 자기검증)

출력 전 각 인사이트에 대해 검증한다: **관찰의 모든 문장이 제목의 주장을 지지하는가?**

- 제목이 "신뢰성이 무너졌다"인데 관찰에 아키텍처 경쟁·도입 추진 팩트가 섞여 있으면 실패.
- 한 건의 뉴스에서만 나온 주장을 제목으로 삼고 무관 팩트로 채우지 않는다 (4U의 Unique 위반).
- 실패 시 둘 중 하나: (a) 제목을 팩트들이 실제로 공유하는 패턴으로 재구성, (b) 제목을 지지하지 않는 팩트를 관찰·근거에서 제거 (해당 기사는 헤드라인·출처에 남음).
- ❌ 제목 "[카테고리] 신뢰성이 무너졌다" + 관찰에 [경쟁사A] 아키텍처 경쟁, [고객사] 도입 → 제목은 기사 한 건에서만 나옴
- ✅ 제목 "데모 영상이 아니라 배치 실적이 [카테고리] 신뢰 기준이 된다" + 관찰 = 데모 조작 시인 + 실제 도입 추진 → 두 팩트가 제목을 양쪽에서 지지

### 관찰
- 한 줄기로 엮이는 팩트만. 관계 약한 건 빼기.
- **모든 관찰 문장은 제목을 뒷받침해야 한다** (위 정합성 검증).
- 해석·전망 섞지 않음. 팩트만.
- [What happened] 만 쓴다. [Why it matters]는 함의 섹션으로.

### 자사 함의 (implication — 자사 맥락 교차 + 함의 통합)
- **구조**: 자사 맥락(narrative/key-events) 먼저 → 그로부터 도출되는 함의.
- 자사 맥락이 없으면 인사이트가 아니라 뉴스 요약. 절대 생략 금지.
- 팩트에서 바로 도출되는 현실만. 희망적 추론 금지.
- **경쟁 구도**: "경쟁사는 이미 X했다. 자사는 아직 Y다."
- **확인 시점**: "6월 8일 간담회 / 이번 분기 / Automate 2026에서 드러난다."
- [자사 맥락 제시 + Why it matters + 확인 대상] 구조.

### 카테고리 요약 (commentary, not just facts — newsletter-management)
- **전부 산문으로 쓴다. 헤드라인 나열 줄을 섞지 않는다.** 그 카테고리에 들어온 기사 전부(tier1 facts + tier2_headlines)를 서술 안에 자연스럽게 녹인다. 독자는 카테고리 요약만 읽고 그날 그 분야 소식을 다 훑었다고 느껴야 한다.
- **주제별로 문단을 나눈다 (핵심).** 한 카테고리 안에서도 소주제(예: 진영 선택 / 자본 유입 / 기술·시연 / 국내 동향)별로 묶어 **주제가 바뀌면 빈 줄(`\n\n`)로 문단을 분리**한다. 기사 많은 카테고리는 3~5문단으로 나눠 읽기 좋게. 한 덩어리로 뭉치지 않는다. format.py가 문단을 보존해 넉넉한 간격으로 렌더한다.
- **"이 밖에 A, B, C, D … 등도 주목됐다" 식 나열 꼬리표 금지.** 비중 낮은 건들도 해당 소주제 문단 안에 한 구절로 자연스럽게 엮는다: "[소주제] 쪽에서는 [A사]의 [동향]과 [B사]의 [동향]이 이어졌다 [13][14]." 회사·사건을 무성의하게 나열하고 번호만 붙이는 문장을 만들지 말 것.
- 같은 사건 복수 보도는 한 문장으로 묶는다. **매체명 나열 금지**.
- key_stakeholder 관련 건이 있으면 첫 문단 선두에.
- "What happened + Why it matters + Your take" 공식 준수.
- **수치 표기 주의: `$1.4B`, `$85M` 같은 달러 금액의 `$`·숫자를 절대 누락하지 말 것.** JSON 파일은 반드시 Write 툴로 저장한다 — bash heredoc(`cat <<EOF`)으로 쓰면 `$1`·`$85`가 셸 변수로 확장돼 사라진다.

## JSON 출력 스키마

```json
{
  "date": "YYYY-MM-DD",
  "tldr": "관통 서사 + 팩트 + 자사 함의",
  "company_glossary": [
    {"name": "[비주류 기업명]", "desc": "[국가] [주력 분야] [업태], [설립연도] 설립"},
    {"name": "[비주류 기업명 2]", "desc": "[국가] [주력 분야] [업태]"}
  ],
  "headlines": [
    {"text": "[경쟁사A] [제품/사업 내용] [수치/결과]", "ref": "a3f9c1", "group": "[경쟁사A]"},
    {"text": "[카테고리 기사 제목 — 한국어 번역]", "ref": "a7b2e4", "group": "[category_id]"}
  ],
  "insights": [
    {
      "title": "[패턴/발견] — [자사 구체 함의 + 시점]",
      "observation": "외부 팩트 2~4문장 (해석 없이 팩트만)",
      "implication": "[수치 선두] → [~라는 뜻이다] → [자사 함의]. 자사 맥락 통합.",
      "derived_from_self_context": ["key-events:[주요 이벤트]", "narrative:positioning"],
      "facts": [
        {"text": "한국어 팩트", "ref": "a3f9c1", "category": "카테고리"}
      ]
    }
  ],
  "category_summary": [
    {"category_id": "...", "category_name": "...",
     "summary": "해당 카테고리 전 기사 커버 — 분량은 기사 수에 비례 (건수·비율 언급 금지: '오늘 X 카테고리는 N건' 패턴 절대 사용 금지)"}
  ],
  "landscape_update_points": [
    {"competitor": "[경쟁사 표기]", "dimension": "ai_sw", "new_info": "[기준선에 없는 새 동향 1문장]", "source": "[매체명]", "date": "YYYY-MM-DD"},
    {"competitor": "[경쟁사 표기]", "dimension": "deployment", "new_info": "[기준선에 없는 새 동향 1문장]", "source": "[매체명]", "date": "YYYY-MM-DD"}
  ]
}
```

- **`ref` 필드 (필수)**: 모든 출처 항목(headlines·facts)은 입력 facts의 `id` 값을 `ref`로 출력한다. **URL·매체명·날짜를 직접 복사하지 않는다** — resolve-refs.py가 ref로 결정론적 조인. ref를 모르는 항목(입력에 없는 기사)만 `source_name`/`source_date`를 직접 쓴다.
- **출력 최소화 (latency 결정 요인)**: 위 스키마에 없는 필드(`all_sources`, `coverage_table`, `category_summary[].sources`, `color` 등)는 **출력하지 않는다**. 전체 출처 목록·커버리지 집계는 format.py가 newsletter-facts에서 결정론적으로 백필한다. LLM 출력 토큰 = 생성 시간이므로, 지능이 필요한 필드(tldr·insights·headlines 선별·category_summary 서술·glossary·landscape_update_points)만 출력한다.
- `headlines`:
  - **일간** (`metadata.hours` ≤ 24): 30초 스캔용 10~15 bullet. 최고 관련성 위주 큐레이션.
  - **주간** (`metadata.hours` > 24): **수집된 전체 unique 기사를 모두 포함. 캡 없음.** 경쟁사별·카테고리별 그룹핑 유지. 동일 사건 복수 보도는 대표 1건만. 일별 서브그룹 불필요 — 카테고리 그룹핑으로 충분. 주간 운용 기준: 중복 제거 후 남은 것은 전부 포함.
  - format.py가 TL;DR과 인사이트 사이에 렌더링.
- `derived_from_self_context`: 어떤 자사 맥락 소스에서 함의 도출했는지 추적 (감사 추적용).
- `uncategorized` facts 중 `relevance_score ≥ 4` 또는 `tier: 1` 기사는 헤드라인·인사이트 소재 후보로도 검토한다. 미분류 ≠ 무관. (예: "[대기업] 대규모 [핵심 인프라] 투자" 단독 보도가 미분류로 누락되던 사례)

## 출력 후 자가 검증 (필수)

briefing JSON 저장 후 반드시 실행한다:

```bash
python3 scripts/newsletter/check-coverage-gaps.py {DATE}
```

- **exit 0** (`✓ 커버리지 갭 없음`) → 완료. 그대로 종료.
- **exit 1** → stdout의 갭 목록(`[{"category": ..., "headline": ...}]`)을 읽고, **중요한** 갭 기사를
  해당 카테고리의 `category_summary[].summary` 산문에 1문장 이내로 반영한다. 전체 재생성 금지 —
  `category_summary`만 수정. 재검증 1회면 충분.
- 사소한 미커버 기사를 산문에 욱여넣지 않아도 된다 — format.py 가 카테고리별로 미인용 수집
  기사를 "이 밖에 ~ 등도 주목됐다 [n]" 으로 자동 쓸어담아 모든 출처번호가 본문에 등장하게
  한다. 합성기는 **중요한 것만 잘 서술**하고, 나머지 sweep 은 렌더러에 맡긴다.

## 사실·문장 단위 인용 (필수)

`insights`(observation·implication·근거)와 `category_summary` 산문의 **모든 사실 주장은
근거 기사의 `ref`(출처 id)로 뒷받침**되어야 한다. format.py 의 attach_inline_refs 가 ref 를
인라인 `[n]` 으로 렌더하므로, 산문은 근거 기사의 제목 키워드(회사명·제품명·수치)를 실제로
포함해 매칭이 걸리도록 쓴다. 키워드 없이 추상적으로만 서술하면 인용이 안 붙는다 —
"한 문장 = 적어도 한 근거" 를 지키고, 근거 없는 단정은 삭제한다.
