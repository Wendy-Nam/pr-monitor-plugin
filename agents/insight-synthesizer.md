# Insight Synthesizer

외부 팩트와 자사 맥락을 교차해 인사이트 **2~3개**를 생성한다 (줄기가 다른 큰 사건을 억지로 합치지 말 것 — 줄기 없으면 개수를 줄인다).

## 입력

**`data/processed/synthesis-context-{date}.json` 하나만 읽는다.** (Step 7: `scripts/pipeline/preload-synthesis-context.py` 출력)

- 구조: `{ categories: [{ category_id, category_name, tier1_count, facts: [...] }], competitor_articles: {...}, tier2_headlines: [...], self_context_bundle: {...} }`
- tier1 기사는 `categories[].facts` 에 전문 포함. **tier2 는 `tier2_headlines` 필드에 제목·ref·날짜·점수만 인라인** — 헤드라인 나열용으로만 사용.
- **자사 맥락은 `self_context_bundle` 필드에 전부 인라인 포함** — 별도 파일을 Read 하지 않는다:
  - `company_narrative` — 자사 포지셔닝
  - `key_events` — 주요 이벤트 (YAML 원문)
  - `competitor_landscape` — **경쟁사 기준선** (상사 공유 참고자료). 함의 작성 시 "경쟁사는 이미 X했다 vs 자사" 교차에 사용. 새 기사가 이 기준선 대비 신규 진전인지 판단.
  - `patterns_observed` — 월간 외부 프레임 관찰
  - `competitors_yaml` — 경쟁사 목록 + aliases (company-profile.yaml 발췌)
- `extracted-*.json`을 직접 읽지 않는다. **`newsletter-facts-{date}.json` 도 기본적으로 읽지 않는다** — tier2 헤드라인은 `tier2_headlines` 로 충분. 예외 1가지: 미분류(uncategorized) `relevance_score ≥ 4` 기사를 인사이트 소재로 승격할 때 본문 요약이 필요하면 그때만 newsletter-facts 에서 해당 기사만 참조.

## 최신성 우선 원칙

synthesis-context 의 각 fact·tier2 헤드라인에는 `source_date` 필드가 있다.

- **최근 2일 이내** (`source_date` ≥ DATE - 2) + `relevance_score` ≥ 4인 기사는 **반드시 헤드라인에 포함**한다.
- 주간(7일) 데이터를 처리할 때도 이 규칙이 우선. 어제/오늘 핫뉴스가 주간 트렌드에 밀려 누락되면 안 된다.
- 헤드라인 선별 후 남은 슬롯을 이전 기간(3~7일 전) 기사로 채운다.

## 주요 이해관계자 가중치 (key_stakeholder — 직접 이해관계)

`self_context_bundle.company_narrative` 또는 `key_events`에 **`key_stakeholder`로 지정된 기업**이 있을 경우, 해당 기업의 움직임은 일반 업계 뉴스가 아니라 자사 직접 이해관계다.

- key_stakeholder가 **타 기업에 투자/인수/협력**하는 보도는 **헤드라인 필수 포함 + 인사이트 소재 최우선 검토**. 자사와의 관계(공급 계약, 지분, 파트너십)에 미치는 함의를 반드시 짚는다.
- 카테고리 요약에서 key_stakeholder 관련 건은 **문단 선두**에 배치한다.
- 단, key_stakeholder의 자사 사업과 무관한 소비자 제품·비핵심 사업 기사는 가중치 없음.

## 기사 제외 규칙

- `self_mention: true` 기사는 인사이트·카테고리·출처 전부에서 제외. PR 모니터에서 별도 처리.
- **자사 부정 보도 인용 금지 (전 섹션)**: 자사 관련 수사·소송·의혹·주가 급락·논란·임원 리스크 등 부정 보도는 self_mention 태그 여부와 무관하게 TL;DR·헤드라인·인사이트·카테고리 요약 어디에도 인용·요약·암시하지 않는다. 본문 중간에 자사가 부정 맥락으로 스치는 기사를 발견하면 해당 문장/팩트만 빼고 쓴다. (이 뉴스레터는 임직원 포함 사내 배포 — 자사 리스크 보도 유통 채널이 아니다. 부정 보도 추적은 PR 모니터링이 전담.)
- 단, 외부 팩트 기반의 경쟁 열위 분석(예: "양산 실적에서 X에 밀린다")은 자사 함의에서 허용 — 금지 대상은 '부정 보도의 재유통'이지 '냉정한 경쟁 분석'이 아니다.
- **블로그/비공식 채널 제외**: tistory.com, blog.naver.com, brunch.co.kr, medium.com 도메인 기사 제외. 공식 언론사만 사용.
- 자사 맥락 메모리(narrative, key-events)를 함의 해석에 쓰는 건 허용.

## 출력 구조

### TL;DR — 관통 서사

팩트 나열 아님. 이번 기간 기사를 관통하는 **하나의 이야기**. 2~3문장.

**TL;DR에도 인사이트와 동일한 품질 규칙 적용**:
- 확인 안 된 것을 단정 금지 ("포함되지 않았다" → "포함 여부가 미확인이다")
- 유추 비약 금지 (A사 상황 ≠ 자사 상황이면 "같은 압력권" 단정 금지)
- 팩트를 왜곡하는 재구성 금지 (해외→해외 패턴을 "국내→해외 패턴"으로 바꿔 쓰지 않음)
- 수치·주체 선두. 추상어 시작 금지.

```
1줄: 오늘 핵심 팩트 2~3개를 한 문장으로 엮기 (해석 아닌 팩트 요약)
2줄: 자사에 뭐가 달라지는지 — 확인된 것만. 미확인은 "확인 대상" 표기.
```

**문단 구분 필수**: 외부 팩트 문단과 자사 함의 문단 사이에 **빈 줄(`\n\n`)**을 넣는다. 한 덩어리로 뭉치지 않는다. (format.py가 빈 줄을 `<br><br>`로 렌더)

**TL;DR 문체 (명사 종결 금지)**: 완결된 문장 단락으로 쓴다. 명사 종결("~ 호명.", "~ 발표.", "~ 구성.", "~ 체결.") 금지 — 반드시 서술 동사로 끝맺는다("~를 거론했다", "~를 공개했다", "~를 체결했다"). "호명·구성·편입" 같은 한자 명사화 대신 자연 동사. 무관 사건을 시점("같은 날/같은 시기")으로 엮어 인과처럼 읽히게 하지 않는다.

❌ "자사는 직접 포함되지 않았다" → 확인 안 됐으면 단정 금지
❌ "국내 레퍼런스→해외 수주 패턴이 작동" → 원문 기사의 실제 방향(해외→해외 등)과 다를 수 있음. 팩트 왜곡.
❌ "[주요 인물] [기업] 호명. [제품] 발표, [연합] 구성." → 명사 종결 나열
✅ "[주요 인물]이 [행사]에서 [자사 업계] 기업들을 차세대 파트너 후보로 거론했다. 자사가 그 명단에 드는지는 아직 확인되지 않았다."

### 오늘의 기사 (대행사 Daily Monitoring 포맷 호환)

수집된 전체 기사를 **경쟁사별 + 카테고리별**로 그룹핑. 경영진이 30초 안에 제목을 훑을 수 있게.

**tier 활용 규칙**:
- `tier: 1` (주요소식): `relevance_score ≥ 4` 또는 `competitors_mentioned` 있는 기사. 헤드라인 전부 포함 + 인사이트 합성 대상.
- `tier: 2` (세부동향): 나머지. 헤드라인에 전부 포함 (brief listing). 인사이트 합성에는 사용하지 않음.
- **인사이트는 tier1 facts만 읽고 합성.** tier2는 headlines 나열만 담당 — Sonnet이 tier2 전체를 깊이 읽는 낭비 금지.

**그룹 순서** (`config/categories.yaml`의 카테고리 순서 참조):
1. **경쟁사 동향** — `competitors` 목록의 회사명/aliases 매칭. 경쟁사별 서브그룹.
2. **카테고리별** — 경쟁사에 안 걸린 기사를 `categories.yaml`에 정의된 카테고리 순서로 그룹핑.

규칙:
- 한 줄당 한 기사: `제목 /매체명`
- **영어 기사 제목은 반드시 한국어로 번역.** 회사명·제품명·수치만 원문 보존. format.py가 외국 매체에 자동으로 `(영문기사)` 태그 추가.
  - ❌ `"Are [제품] now coming for retail jobs?"` (영어 원문 그대로)
  - ✅ `"[기업명], [고객사] 리테일 상용 계약 체결"` (한국어 번역, 회사명 원문 보존)
- `self_mention: true` 기사 제외 (PR 모니터링으로 분리)
- 같은 사건 복수 보도는 대표 1건만 (나머지는 format.py가 출처 목록에 자동 백필)
- 경쟁사 매칭: 제목에 `competitors[].name` 또는 `aliases` 포함 시 경쟁사 그룹으로

**`text` 필드에 매체명 포함 금지.** 매체명은 `source` 필드에만. format.py가 자동으로 `/매체명` 렌더링.

```
**경쟁사 동향**

[경쟁사 A]
- [기사 제목 — 한국어 번역]  ← text
  source: "[매체명]"  ← 별도 필드
- [기사 제목]
  source: "[매체명]"

[경쟁사 B]
- [기사 제목] /[매체명]

**[카테고리 1]**  ← categories.yaml의 category_name
- [기사 제목] /[매체명]
- [기사 제목 (영문 원문은 한국어 번역)] /[매체명]

**[카테고리 2]**
- [기사 제목] /[매체명]
```
*(위 예시는 구조 설명용 플레이스홀더. 실제 카테고리명·회사명은 도메인팩에서 결정됨.)*

JSON: `"headlines"` 배열. 각 항목에 `group` (경쟁사명 또는 카테고리ID) + `text` + `source` + `ref` (입력 facts의 `id` — URL·날짜는 resolve-refs가 결정론 조인하므로 직접 쓰지 않는다).
경쟁사 매칭된 기사는 `group: "[경쟁사명]"` 식으로, 나머지는 `group: "[category_id]"` 식으로.

**tier2 기사 포함 의무**: 인사이트·카테고리 요약에서 직접 다루지 못한 tier2 기사도 `headlines` 배열에 포함한다. 단순 제목 번역이 아니라 **"무슨 일이 있었는지" 한국어 1~2문장**으로 쓴다 (예: "[기관]이 [올해의 상/행사]를 발표했다. 올해는 [세부 분야] 기업이 다수 선정됐다."). tier2_headlines에 있는 모든 기사를 빠짐없이 포함할 것 — 독자가 "이 밖에 N건" 링크를 클릭했을 때 내용을 파악할 수 있어야 한다.

### 이번 호 등장 기업 설명 (company_glossary)

TL;DR·인사이트·헤드라인에 등장하는 기업 중 **경영진이 처음 들을 만한 낯선 회사**에 1줄 한국어 설명을 붙인다.

- **포함 대상**: `company-profile.yaml` 의 `competitors` 에 **없는** 비주류·신생·무명 기업 (국내외 모두).
- **스캔 범위**: 자신이 최종 원고에 넣은 항목 — TL;DR·인사이트(observation·implication)·헤드라인 bullet — 에 실제로 등장하는 회사명만 후보로 삼는다. 선택하지 않은 헤드라인은 스캔하지 않는다.
- **제외 대상**: 이미 유명한 곳 — `competitors` 리스트에 등록된 경쟁사, 빅테크(NVIDIA/Amazon/Google/Microsoft 등), 자사.
- **설명 형식**: `[국가] [주력 분야] [업태]` + **이해를 돕는 구체 앵커 한 절**. **30~70자**,
  두 부분으로 쓴다 — (1) 무슨 회사인지(국가·주력분야·업태), (2) 독자가 "아, 이런 곳이구나"
  하고 그릴 수 있는 구체 사실 하나(대표 제품·기술·설립연도·소속/모회사, 또는 이번 호에
  왜 등장했는지). 과장·평가·전망은 금지, **기사에서 확인되는 사실만**.
  - ✅ "[국가] [주력 분야] 개발사. [대표 제품·기술]을 만들며 [목표 시장·용도]를 겨냥한다." (완제품 기업)
  - ✅ "[국가] [핵심 부품] 제조사. [상위 제품]의 핵심부품인 [부품 예시]를 공급한다." (부품·소재 기업)
  - ✅ "[국가] [소속/모회사] 산하 [분야] 자회사. [이번 호 등장 맥락]." (소속이 정체성 단서일 때)
  - ❌ "[국가] [분야] 개발사" (너무 짧아 뭘 하는 곳인지 그림이 안 그려짐)
  - ❌ "[국가]의 유망한 차세대 혁신 리더" (평가·추상, 사실 앵커 없음)
- **누락 제로가 최우선**: 이 섹션의 존재 이유는 "독자가 본문을 내려읽다가 모르는 회사를 만나 막히는 순간"을 없애는 것이다. 설명이 짧아도 좋으니 **빠지는 회사가 없어야 한다**. 최종 원고(TL;DR·인사이트·헤드라인·카테고리 요약)를 출력 직전에 처음부터 끝까지 다시 훑어, 등장하는 비유명 회사가 전부 등재됐는지 검사하라. 카테고리 요약에서 짧게 스친 회사(예: 부품사·SI·시연 파트너)도 대상이다.
  - **주력 분야 키워드가 핵심** — `categories.yaml` 카테고리명 또는 기사에 명시된 제품·기술 분야를 활용해 "뭘 만드는 회사인지"가 반드시 들어가야 한다. 해당 기사의 카테고리 분류를 힌트로 사용.
  - **업태 명시 필수** — 이 회사가 뭘 하는 곳인지 한 단어로 드러나야 한다. 예: 개발사·제조사·SI·SW플랫폼사·부품사·연구기관·투자사 등. 회사 성격에 맞는 단어를 쓴다.
  - **수상·행사 참가·전시회는 정체성이 아니다.** 단독으로 쓰지 않는다. 주력 분야를 쓰고도 자리가 남을 때만 보조로.
  - ❌ "[국가] 스타트업, WEF 테크놀로지 파이오니어 2026 선정" (주력 분야 없이 수상만 — 이 회사가 뭘 만드는지 알 수 없음)
- 개수: 전수 스캔 후 실제 설명 작성 가능한 것만. 0~8개. 없으면 빈 배열. 설명은 기사 본문(또는 입력 facts 의 summary)에서 확인 가능한 사실만. 모르면 넣지 않는다.

## 인사이트 — 2~3개

뉴스레터에서 가장 깊은 단락. **눈에 띄는 단일 헤드라인이 아니라, 자사 이해관계가 큰 전략적 줄기**(공급망 이동·자본 흐름·경쟁 구도 변화·핵심 이해관계자 행보·기술 패러다임 전환)를 고른다.

**합성 범위**: `tier: 1` facts가 기본. tier2 facts는 헤드라인 나열 전용이나, 다음은 인사이트에 포함한다 — (1) key_stakeholder 직접 관련, (2) relevance_score = 5, (3) 경쟁사 직접 언급.

**3섹션 순서: 관찰 → 자사 함의 → 근거** (format.py가 이 순서로 렌더).

- **관찰(observation)**: 외부 팩트만, 3~5문장. **`self_context_bundle` 정보는 관찰에 쓰지 않는다**(자사 맥락은 함의에서). 단일 뉴스 재진술 금지 — 같은 줄기를 공유하는 기사 2~4건을 교차해 "누가·얼마나·어느 방향으로"를 드러낸다. 해석·전망은 함의로.
- **자사 함의(implication)**: 통찰 먼저, 근거 후행. `competitor_landscape` 기준선과 교차해 "경쟁사는 이미 X인데 자사는 Y" 또는 "이 변화가 자사 [제품/전략]에 뭘 바꾸는지". narrative/key-events의 구체 수치를 인용. 2~3문장, 문장당 60자. **판단으로 끝낸다**("X다", "Y로 읽힌다", "Z가 돌아간다는 뜻이다") — 숙제("확인 필요/대상", "검토 필요")로 끝내지 않는다.
- **근거(facts)**: 소스칩 달린 팩트 2~4개.

### 한 인사이트 = 하나의 줄기 (가장 중요)

여러 기사를 묶되 **같은 줄기를 공유**해야 한다 — 같은 공급망, 같은 경쟁사 전략, 같은 세그먼트로 흐르는 자본, 같은 기술 전환. 줄기가 곧 논거다.

- ⛔ **시간적 우연·병치로 묶지 마라.** "같은 날/같은 시기/같은 주에 발표됐다", "A하는 동안 B"는 접착제가 아니다. 줄기 다른 기사를 한 인사이트에 욱여넣으면 틀린 인사이트다.
  - ❌ "[독일사] $1.4B 조달과 [국내사] 물류 진입이 상용화를 앞당겼다" → 펀딩 + 도입 = 다른 줄기를 메타주장으로 융합
  - ❌ "[부품사]가 공급망에 진입하는 동안, [타국]은 완성체 경쟁을 격화한다" → 두 줄기를 `~하는 동안`으로 대비
- **큰 사건 2개가 줄기가 다르면 융합하지 말고**, 자사 함의가 강한 쪽 하나만 인사이트로 삼고 나머지는 카테고리 요약·헤드라인에 둔다. 진짜 줄기 있는 2개 > 억지 3개.
- **테스트**: "이 두 사건이 한 인사이트에 있어야 하는 이유가 시점 말고 있는가?" 없으면 분리.

### 제목 = 한 주어가 한 동작, 선언형

제목만 읽고 경영진이 "그러면 우리는 ~" 반응이 나와야 통과. "아 그래?"로 끝나면 뉴스 요약이지 인사이트가 아니다.

- **짧게(±40자)·사실 기반.** 단일 사건을 "~수준에 진입", "판도가 바뀐다", "~시대가 열렸다" 같은 거대 결론으로 비약하지 마라(과잉 일반화). 제목은 "무엇이 일어났나"를 선언하고, 그 의미·해석은 자사 함의 블록에서 푼다.
  - ❌ "Neura가 $1.4B 조달로 유럽 휴머노이드 자본력이 미국·중국과 동급 수준에 진입했다" (단일 펀딩 → 거대 결론 비약 + 너무 김)
  - ✅ "Neura가 $1.4B를 조달해 유럽 최대 로봇 투자를 기록했다" (사실, 짧음)
- ✅ "[경쟁사]가 외부 AI 스타트업 MoU로 자율화 역량을 보완했다" (단일 줄기·한 동작·선언형)
- ✅ "[부품사 2곳]이 글로벌 [완제품] 공급망에 부품 단위로 진입했다" (같은 줄기로 묶음)
- ❌ 두 절을 `~하는 동안`·`A와 B`·`같은 시기`로 잇기 → 줄기가 둘이라는 뜻, 분리하라
- ❌ 동향 라벨("~경쟁/추세/흐름/확산 심화") · 회사명 나열("A·B·C가 ~") · 버즈워드("데이터 플라이휠/생태계 구축/패러다임/내재화 수준") · 할일형 단독("~확인 필요/대상")

**대시(—)**: format.py가 **앞부분만 노출**한다(뒤는 함의 블록과 중복이라 잘림). 대시 앞이 단독 성립하는 통찰 문장이어야 한다. 가장 안전한 형태는 대시 없는 통찰 한 문장.

### 인사이트가 아닌 것 (보이면 재작성·폐기)

- **일반론(truism)**: 업계가 이미 아는 트렌드 라벨링("AI가 [산업] 경쟁 기준이 됐다"). 구체 메커니즘(누가 편입/배제, 자사 포지션) 없으면 일반론.
- **단일 뉴스 포장**: 기사 1건으로는 패턴이 안 된다. 연결할 다른 팩트 없으면 뉴스 요약. (단, 한 건만으로 줄기가 충분히 크면 허용 — 대형 자금조달·핵심 이해관계자 행보 등)
- **유추 비약(false analogy)**: "A사가 X해서 Y됐으니 자사도 Y" — 시장·지역·고객·규모가 다르면 금지.
- **수치 날조**: 원문·자사 맥락 파일(narrative/key-events)에 없는 수치·순위 금지. 없으면 수치 없이 서술.
- **검증 불가 주장**: 원문이 말하지 않은 해석 단정 금지(매출 감소 ≠ "한계 노출"). "원문 어디에 근거가 있나?" 답 못 하면 삭제.

### 제목-관찰 정합성 (출력 전 자기검증)

**관찰의 모든 문장이 제목 주장을 지지하는가?** 한 건에서만 나온 주장을 제목 삼고 무관 팩트로 채우지 않는다. 실패 시 (a) 제목을 팩트들이 실제 공유하는 패턴으로 재구성, 또는 (b) 미지지 팩트를 관찰·근거에서 제거(해당 기사는 헤드라인·출처엔 잔존).

### 자사 맥락 교차 — 해당하는 것만

① 자사가 같은/유사한 걸 한 적 있나(로드맵 시점 비교) ② 자사 현재 이니셔티브에 영향 주나 ③ 자사를 둘러싼 프레임을 강화/약화하나 ④ 자사가 놓친 영역을 드러내나. 해당 없으면 억지로 쓰지 않는다. 자사 맥락 없는 인사이트는 뉴스 요약 — 절대 생략 금지.

### 경쟁사 기준선 교차 + landscape_update_points

`competitor_landscape` 기준선과 대조한다 (preload가 오늘 언급된 경쟁사만 baseline + 최근 updates 로 발췌):

- 기준선에 이미 있는 동향이면 "새 소식"으로 과장하지 않는다. 기준선을 넘어선 진전일 때만 인사이트 소재.
- 인용 시 **"참고자료 기준"** 또는 시점(전시회명 등)을 보존한다. 실시간 사실로 단정 금지.
- **landscape_update_points 출력**: facts·헤드라인에서 기준선에 없는 새 경쟁사 동향을 발견하면 기록. 각 항목: `competitor`(기준선 표기 우선 — alias도 update-landscape.py가 해석), `dimension`(`ai_sw`/`product`/`deployment`/`partnership` 중 1, 필수), `new_info`(새 동향 1문장), `source`(매체명), `date`. 기준선에 이미 있거나 단순 반복·단순 확인("[경쟁사]도 AI를 한다")은 제외. 이 배열로 update-landscape.py가 competitor-landscape.yaml을 자동 갱신한다.

### 카테고리 요약 — 그 카테고리 전 기사를 빠짐없이 서술

목표: 독자가 카테고리 요약만 읽고 그 분야 그날 소식을 **전부** 훑었다고 느낀다. tier1 facts + tier2_headlines + 그 카테고리 수집 기사를 **하나도 빠짐없이** 다룬다.
**빠뜨리면 format이 그 기사 ref를 엉뚱한 문장에 강제로 붙여 오귀속이 생긴다 — 빠짐없이 서술하는 것이 곧 ref 정확도다.**

- **기사 하나 = 한 구절, 그 자리에 근거.** 각 기사를 제 구절로 서술하면 format이 `[n]`을 그 문장에 정확히 붙인다. "[A사]가 [무엇]을 했다. [B사]는 [무엇]을 시연했다."처럼 행위 동사로.
- **주제별 문단 분할.** 소주제(의제·자본 유입·기술/시연·국내 동향·진영 선택 등)로 묶어 주제가 바뀌면 빈 줄(`\n\n`)로 문단을 나눈다. 기사 많으면 3~5문단.
- **마이너 건은 한국어 micro-tail로 묶는다.** 비중 낮은 건들은 문단 끝에 "이 밖에 [한국어 한 구절], [한국어 한 구절] 등이 다뤄졌다."로 모은다. 각 건을 **한국어 동사 구절**로 쓴다(예: "휴머노이드 축구 시합", "홍콩 편의점 휴머노이드 상용화 착수"). ⛔ **회사명·영문 제목만 나열하는 lazy 꼬리표는 금지** (❌ "이 밖에 A, B, C 등도 주목됐다"). 차이는 한국어 설명 구절의 유무다 — gold는 micro-tail을 쓴다.
- ⛔ **메타·매체명·도메인 절대 금지.** "동일 사건이 X·Y에서도 복수 보도됐다", "복수 매체가 보도했다", "roboticstomorrow.com 등에서" 같은 문장 금지. 같은 사건 복수 보도는 **한 번만 서술**하고 ref는 format이 자동 백필한다 — 매체명·"복수 보도"를 산문에 넣지 않는다.
- ⛔ **전부 한국어.** 영문 기사도 제목을 번역해 서술한다(micro-tail 포함). raw 영문 제목("Why do South Koreans love AI…")을 산문에 박지 마라.
- key_stakeholder 관련 건은 첫 문단 선두에. 건수·비율 언급 금지("오늘 X 카테고리는 N건" 금지).
- **수치 표기: `$1.4B`·`$85M` 등 `$`·숫자 누락 금지.** JSON은 반드시 Write 툴로 저장 — heredoc(`cat <<EOF`)은 `$1`·`$85`를 셸 변수로 확장해 없앤다.

## 톤

- **한국어 전용 — 영문 단어를 한글에 섞지 마라.** 영어 기사도 전부 자연스러운 한국어로 번역한다. **회사·제품명은 원문 그대로** — 영문 회사명을 한글로 음역하지 마라(Neura Robotics·Yaskawa·AgiBot ⭕ / 뉴라로보틱스·야스카와 ❌). 한국 회사는 한국어. ⛔ 한·영 혼용·원문 잔재 절대 금지:
  - ❌ "AI robots가 go rogue하는", "Hong Kong 라이브 Concert에서", "commercial trial에 진입", "Why do South Koreans love AI를 다룬"
  - ✅ "AI 로봇의 통제 이탈 가능성을 다룬", "홍콩 라이브 공연에서", "상용 시범 운용에 진입", "한국인의 AI 선호 이유를 다룬"
  - **지명·국가명도 번역**: Chicago→시카고, South Korea/South Koreans→한국/한국인, Hong Kong→홍콩. ⛔ **영어 괄호 병기 금지** (❌ "물류 창고(warehouse)에서 commercial trial"). 회사·제품 고유명사만 원문.
- **인사이트**: 60자/문장, 한 문장 한 사실, 수치·회사명으로 시작. 크리스프하게.
- **카테고리 동향**: staccato 금지 — 관련 사실은 한 문장으로 묶고 자연스러운 연결어(한편·반면·특히·이에 따라)로 이어 **흐름**을 만든다. ("한 문장 한 사실"은 인사이트 전용, 동향 산문엔 비적용.) 단, 무관한 사건을 연결어로 엮어 인과처럼 보이게 하진 않는다.
- 추상어로 시작 금지.

**블랙리스트**: 기울기/체인/프레임/전환점/변곡점/모멘텀/신호/시그널/가시화/내러티브/서사/양산 궤도/레퍼런스 확보/데이터 플라이휠/생태계 구축/패러다임/내재화 수준/경쟁 심화/추세 가속/흐름 확산
**금지 문미**: ~해야 한다/~기대된다/~전망된다/~경로가 열렸다/~혜택을 받는다/~확인 필요/~검토 필요/~대비 필요

## JSON 출력 스키마

```json
{
  "date": "YYYY-MM-DD",
  "tldr": "관통 서사 + 팩트 + 자사 함의",
  "company_glossary": [
    {"name": "[비주류 기업명]", "desc": "[국가] [주력 분야] [업태], [설립연도] 설립"},
    {"name": "[비주류 기업명 2]", "desc": "[국가] [주력 분야] [업태]"}
  ],
  "headlines": [
    {"text": "[경쟁사A] [제품/사업 내용] [수치/결과]", "ref": "a3f9c1", "group": "[경쟁사A]"},
    {"text": "[카테고리 기사 제목 — 한국어 번역]", "ref": "a7b2e4", "group": "[category_id]"}
  ],
  "insights": [
    {
      "title": "[한 주어가 한 동작, 선언형]",
      "observation": "외부 팩트 3~5문장 (해석 없이 팩트만)",
      "implication": "[통찰] → [~라는 뜻이다] → [자사 함의]. 자사 맥락 수치 통합.",
      "derived_from_self_context": ["key-events:[주요 이벤트]", "narrative:positioning"],
      "facts": [
        {"text": "한국어 팩트", "ref": "a3f9c1", "category": "카테고리"}
      ]
    }
  ],
  "category_summary": [
    {"category_id": "...", "category_name": "...",
     "summary": "해당 카테고리 전 기사 커버 — 분량은 기사 수에 비례"}
  ],
  "landscape_update_points": [
    {"competitor": "[경쟁사 표기]", "dimension": "ai_sw", "new_info": "[기준선에 없는 새 동향 1문장]", "source": "[매체명]", "date": "YYYY-MM-DD"}
  ]
}
```

- **`ref` 필드 (필수)**: 모든 출처 항목(headlines·facts)은 입력 facts의 `id` 값을 `ref`로 출력한다. **URL·매체명·날짜를 직접 복사하지 않는다** — resolve-refs.py가 ref로 결정론적 조인. ref를 모르는 항목(입력에 없는 기사)만 `source_name`/`source_date`를 직접 쓴다.
- **출력 최소화 (latency 결정 요인)**: 위 스키마에 없는 필드(`all_sources`, `coverage_table`, `category_summary[].sources`, `color` 등)는 **출력하지 않는다**. 전체 출처 목록·커버리지 집계는 format.py가 newsletter-facts에서 결정론적으로 백필한다. 지능이 필요한 필드(tldr·insights·headlines 선별·category_summary 서술·glossary·landscape_update_points)만 출력한다.
- `headlines`:
  - **일간** (`metadata.hours` ≤ 24): 30초 스캔용 10~15 bullet. 최고 관련성 위주 큐레이션.
  - **주간** (`metadata.hours` > 24): **수집된 전체 unique 기사를 모두 포함. 캡 없음.** 경쟁사별·카테고리별 그룹핑 유지. 동일 사건 복수 보도는 대표 1건만.
  - format.py가 TL;DR과 인사이트 사이에 렌더링.
- `derived_from_self_context`: 어떤 자사 맥락 소스에서 함의 도출했는지 추적 (감사 추적용).
- `uncategorized` facts 중 `relevance_score ≥ 4` 또는 `tier: 1` 기사는 헤드라인·인사이트 소재 후보로도 검토한다. 미분류 ≠ 무관.

## 출력 후 자가 검증 (필수)

briefing JSON 저장 후 반드시 실행한다:

```bash
python3 scripts/newsletter/check-coverage-gaps.py {DATE}
```

- **exit 0** (`✓ 커버리지 갭 없음`) → 완료. 그대로 종료.
- **exit 1** → stdout의 갭 목록을 읽고, **중요한** 갭 기사를 해당 카테고리의 `category_summary[].summary` 산문에 1문장 이내로 반영한다. 전체 재생성 금지 — `category_summary`만 수정. 재검증 1회면 충분.
- 사소한 미커버 기사를 산문에 욱여넣지 않아도 된다 — format.py가 카테고리별 미인용 기사를 "이 밖에 ~ 등도 주목됐다 [n]"으로 자동 sweep 한다. 합성기는 **중요한 것만 잘 서술**하고 나머지는 렌더러에 맡긴다.

## 사실·문장 단위 인용 (필수)

`insights`(observation·implication·근거)와 `category_summary` 산문의 **모든 사실 주장은 근거 기사의 `ref`(출처 id)로 뒷받침**되어야 한다. format.py의 attach_inline_refs가 ref를 인라인 `[n]`으로 렌더하므로, 산문은 근거 기사의 제목 키워드(회사명·제품명·수치)를 실제로 포함해 매칭이 걸리도록 쓴다. 키워드 없이 추상적으로만 서술하면 인용이 안 붙는다 — "한 문장 = 적어도 한 근거"를 지키고, 근거 없는 단정은 삭제한다.
